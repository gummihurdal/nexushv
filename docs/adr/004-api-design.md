# ADR-004: API Design Principles

## Status
Accepted

## Context
NexusHV API serves both the React frontend and third-party integrations.

## Decisions

### URL Structure
- All REST endpoints under `/api/`
- WebSocket endpoints under `/ws/`
- Prometheus metrics at `/metrics`
- Health check at `/health`
- API docs at `/api/docs` (Swagger) and `/api/redoc` (ReDoc)

### Authentication
- JWT Bearer tokens for REST API
- Optional auth (some endpoints work without auth for development)
- WebSocket authentication via query parameter or first message

### Versioning
- Currently v2, no URL versioning prefix (simple for now)
- Breaking changes will be introduced with `/api/v3/` prefix
- Feature discovery via `/api/info` endpoint

### Response Format
- All responses are JSON
- Error responses: `{"detail": "error message"}`
- List responses: plain JSON arrays
- Paginated responses: `{"items": [...], "total": N, "page": N}`

### Naming Conventions
- Resource-oriented URLs: `/api/vms/{name}`
- Actions as sub-resources: `/api/vms/{name}/action`
- Batch operations: `/api/batch/{operation}`
- Recommendations: `/api/recommendations/{type}`

### Real-time Data
- WebSocket `/ws/metrics`: Host and VM metrics every 2s
- WebSocket `/ws/events`: Alerts, VM state changes, HA events
- WebSocket `/ws/ai/stream`: Token-by-token AI responses

## Consequences
- Frontend and external tools use the same API
- OpenAPI spec auto-generated by FastAPI
- Easy to add API versioning later if needed
